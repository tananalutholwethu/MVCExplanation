Desktop vs Web

Think of it like this...

Your VB World

Like a WhatsApp call

Continuous connection, stays active

  • Program stays running in memory
  • Variables stay the same between actions
  • Forms stay loaded
  • One user = one program instance

Web World

Like WhatsApp messages

Send request, get response, forget everything. Repeat.

  • Program runs only when needed
  • Variables destroyed after each request
  • Pages rebuilt from scratch each time
  • Many users = same program files
Web is "stateless" - It doesn't remember anything between requests. Nothing is saved unless you store it yourself (using sessions, cookies, or a database)

How Web Actually Works

Every page load is a complete round trip

1. You click
Browser sends HTTP request
2. Web server receives
Apache/Nginx handles it
3. PHP executes
Your code runs
4. Database query
Fetches data
Then everything comes back...
5. Data returns
Result set from DB
6. PHP builds HTML
Creates page markup
7. Server sends
Complete HTML response
8. Browser renders
You see the page
Why this matters: This happens EVERY time you click or load a page. Understanding this cycle is crucial for web development.
But isn't this complicated? Actually, it's simpler than it seems. A desktop app has to be installed on every user's computer, but one web app can work for thousands of people at the same time using the same code. Starting fresh each time also helps prevent memory problems and makes fixing errors easier.

Wait... PHP Creates HTML?

What does that actually mean?

You might be thinking:

"Doesn't HTML already exist in files? How can PHP create it?"

Here's the Simple Truth:

HTML Files Exist

Yes, you write .html or .phtml files with HTML code

These are like templates - the structure is there

PHP Fills In The Blanks

When someone visits your page, PHP sprinkles in the data

It builds the final HTML right before sending it

Your Template File:
<h1>Users</h1>
<p>Hello, [NAME GOES HERE]</p>
PHP runs and fills it in:
<?php $name = "John"; ?>
<h1>Users</h1>
<p>Hello, <?= $name ?></p>
Browser gets pure HTML:
<h1>Users</h1>
<p>Hello, John</p>
The Key Point: You write HTML files. PHP adds the dynamic parts. The browser never sees the PHP - it only sees the finished HTML result.

Variables: VB vs PHP

Same concept, slightly different syntax

VB.NET

' Declare variable
Dim name As String
name = "John"

Dim age As Integer
age = 25

PHP

<?php
// No declaration needed!
$name = "John";

$age = 25;
?>

Key Differences

  • All variables start with $
  • No type declaration (PHP figures it out)
  • Use . for string joining, not &
  • Comments use // or /* */

POST vs GET: What's The Difference?

Two ways to send data to the server

POST

Data hidden in request body

  • Data NOT visible in URL
  • More secure
  • No size limit
  • Cannot save/share the link
Use for:

Login forms, submitting data, passwords, creating/updating records

GET

Data visible in URL

  • Data visible in URL bar
  • Less secure
  • Size limit (~2000 chars)
  • Can save/share the exact link
Use for:

Search queries, filters, pagination, shareable links

Real URL Example:

https://takealot.com/search?query=laptop&category=electronics&min_price=5000

This is GET - all search parameters are in the URL. You can copy this link and send it to someone, and they'll see the exact same search results!

Real World Comparison:

GET Example:

Takealot search - you want to share your search results with a colleague

takealot.com/search?query=monitor
POST Example:

Banking login - you definitely don't want your password in the URL!

nedbank.co.za/login (credentials sent securely in POST)
Security Rule: Never send passwords or sensitive data via GET. Anyone can see the URL - in browser history, server logs, over someone's shoulder!

Understanding MVC

Model - View - Controller

What Is MVC?

A way to keep your code from getting messy

VIEW

What the user sees

  • HTML templates
  • Forms & tables
  • Display only!
Like: Restaurant menu

CONTROLLER

The coordinator

  • Handles requests
  • Business logic
  • Coordinates everything
Like: The waiter

MODEL

Data handler

  • Database queries
  • Data validation
  • Returns data
Like: The kitchen

Why separate these?

  • Organization: Easy to find things
  • Reusability: Use same Model in different places
  • Teamwork: Designer works on Views, you work on Controllers
  • Maintenance: Change one part without breaking others

How URLs Work in MVC

The URL tells your app exactly where to go

Let's Break Down This URL:

http://mvcshell.webdev.luthot/functions/create
functions

Controller Name

Points to: app/Controllers/Functions.php

This is like choosing which waiter handles your order

create

Action Name

Runs method: createAction() in Functions.php

This is like choosing what to order from the menu

Auto-loads View

View File

Displays: app/Views/Functions/create.phtml

The framework automatically knows which view file to show!

The Pattern:

/{controller}/{action}

Becomes...

  • Controller: app/Controllers/{Controller}.php
  • Method: {action}Action()
  • View: app/Views/{Controller}/{action}.phtml

More Examples:

URL Controller File Method View File
/users/index Users.php indexAction() Users/index.phtml
/provinces/create Provinces.php createAction() Provinces/create.phtml
/reports/monthly Reports.php monthlyAction() Reports/monthly.phtml
Pro Tip: The framework does all this routing automatically! You just need to name your files and methods correctly.

The name you'll use to get this value

Like a variable name

The ID of the function to update

See? Every part of the URL becomes accessible to your code!

Using the Parameter in Your Code:

Getting the ID

Province CRUD Module

Time to create our own MVC from scratch

Step 1: Create Province Model

SQL queries belong in the Model - this is the proper way!

Create file: app/Models/Tables/Province.php
Province.php 
<?php
namespace App\Models\Tables;

class Province extends \Core\Model {
    
    public function __construct() {
        $this->tableName = 'province';
        parent::__construct();
    }
    
    /**
     * Get all provinces
     * SQL query lives here in the Model!
     */
    public function getAllProvinces() {
        $query = "SELECT * FROM province ORDER BY province_name";
        return $this->getAll($query);
    }
    
    /**
     * Get single province by ID
     */
    public function getProvinceById($id) {
        $query = "SELECT * FROM province WHERE id = ?";
        return $this->getRow($query, [$id]);
    }
    
    /**
     * Check if province code already exists
     */
    public function codeExists($provinceCode) {
        $query = "SELECT COUNT(*) as count 
                  FROM province 
                  WHERE province_code = ?";
        
            $result = $this->getRow($query, [$provinceCode]);
        }
        
        return $result['count'] > 0;
    }
}

Why Put SQL in the Model?

  • Organization: All database stuff stays in one place
  • Reusability: Other controllers can use the same methods
  • Testability: Easy to test database logic separately
  • Maintenance: Change the query once, works everywhere

Could You Put SQL in the Controller?

Yes, technically you can do this:

// In Controller (not recommended for big projects)
$query = "SELECT * FROM province";
$provinces = $model->getAll($query);

But... for learning and best practices, we're putting SQL in the Model. It's cleaner, more professional, and easier to maintain as your project grows!

Step 2: Create Province Controller

The controller just calls the model's methods - nice and clean!

Create file: app/Controllers/Provinces.php
Provinces.php 
<?php
namespace App\Controllers;

class Provinces extends \Core\Controller {
    
    /**
     * List all provinces
     * URL: /provinces/index or just /provinces
     */
    public function indexAction() {
        $provinceModel = new \App\Models\Tables\Province();
        
        // Look how clean! No SQL here, just call the method
        $this->view->provinces = $provinceModel->getAllProvinces();
        
        // View automatically loads: app/Views/Provinces/index.phtml
    }
    
    /**
     * Show create form and handle submission
     * URL: /provinces/create
     */
    public function createAction() {
        if ($this->getRequest()->isPost()) {
            $params = $this->getAllParams();
            
            $provinceModel = new \App\Models\Tables\Province();
            
            // Check if code already exists
            if ($provinceModel->codeExists($params['code'])) {
                $this->view->error = "Province code already exists!";
                return;
            }
            
            $data = [
                'name' => $params['name'],
                'code' => strtoupper($params['code'])
            ];
            
            $provinceModel->autoExecute(
                'province', 
                $data, 
                DB_AUTOQUERY_INSERT
            );
            
            $this->redirect('/provinces');
        }
        
        // If not POST, just show the form
        // View loads: app/Views/Provinces/create.phtml
    }
    
    /**
     * Edit existing province
     * URL: /provinces/update/id/5
     */
    public function updateAction() {
        // Debug tip: See what's in the URL
        $params = $this->getAllParams();
        // print_debug($params);  // Uncomment to debug!
        
        $id = $params['id'];
        $provinceModel = new \App\Models\Tables\Province();
        
        if ($this->getRequest()->isPost()) {
            // Form was submitted
            $data = [
                'name' => $params['name'],
                'code' => strtoupper($params['code'])
            ];
            
            $provinceModel->autoExecute(
                'province', 
                $data, 
                DB_AUTOQUERY_UPDATE,
                "id = '$id'"
            );
            
            $this->redirect('/provinces');
        }
        
        // Load existing data for the form
        $this->view->province = $provinceModel->getProvinceById($id);
        
        // View loads: app/Views/Provinces/update.phtml
    }
    
    /**
     * Delete a province
     * URL: /provinces/delete/id/5
     */
    public function deleteAction() {
        $params = $this->getAllParams();
        $id = $params['id'];
        
        if ($id) {
            $provinceModel = new \App\Models\Tables\Province();
            $provinceModel->delete("id = '$id'");
        }
        
        $this->redirect('/provinces');
    }
}

What Happens When You Visit These URLs:

/provinces or /provinces/index

‚Üí Runs indexAction() ‚Üí Shows list (index.phtml)

/provinces/create

‚Üí Runs createAction() ‚Üí Shows form (create.phtml)

/provinces/update/id/5

‚Üí Runs updateAction() ‚Üí Gets id=5 from URL ‚Üí Shows form (update.phtml)

/provinces/delete/id/5

‚Üí Runs deleteAction() ‚Üí Gets id=5 from URL ‚Üí Deletes ‚Üí Redirects

See the Benefits? The Controller is clean and readable. All the messy SQL lives in the Model where it belongs. This is professional code structure!
